## Authentication

<Note>
  All endpoints require authentication except `/` (API directory) and `/health` (health check).
</Note>

Add your API key to every request using the `X-API-Key` header:

<CodeGroup>
```bash cURL
curl -H "X-API-Key: YOUR_KEY" \
  https://api.shadowcorp.dev/v1/stats
```

```javascript JavaScript
const response = await fetch(
  'https://api.shadowcorp.dev/v1/stats',
  {
    headers: {
      'X-API-Key': 'YOUR_KEY'
    }
  }
);
```

```python Python
import requests

response = requests.get(
    'https://api.shadowcorp.dev/v1/stats',
    headers={'X-API-Key': 'YOUR_KEY'}
)
```

```go Go
req, _ := http.NewRequest("GET", 
    "https://api.shadowcorp.dev/v1/stats", nil)
req.Header.Add("X-API-Key", "YOUR_KEY")

client := &http.Client{}
resp, _ := client.Do(req)
```
</CodeGroup>

## Pagination

List endpoints support pagination through query parameters:

<ParamField query="limit" type="integer" default="50">
  Number of results to return per request. Maximum: 100
</ParamField>

<ParamField query="offset" type="integer" default="0">
  Number of results to skip. Use for paginating through large result sets.
</ParamField>

**Example:**

```bash
# Get first 50 transactions
curl -H "X-API-Key: YOUR_KEY" \
  "https://api.shadowcorp.dev/v1/transactions?limit=50&offset=0"

# Get next 50 transactions
curl -H "X-API-Key: YOUR_KEY" \
  "https://api.shadowcorp.dev/v1/transactions?limit=50&offset=50"
```

<Tip>
  For optimal performance, use the maximum limit of 100 when fetching large datasets, then paginate through results using the offset parameter.
</Tip>

## Response Format

All successful responses return JSON with a `200 OK` status code:

```json
{
  "data": [...],
  "metadata": {
    "limit": 50,
    "offset": 0,
    "total": 1234
  }
}
```

For single-resource endpoints, the response is a JSON object:

```json
{
  "id": "example.near",
  "balance": "1000000000000000000000000",
  "staked": "500000000000000000000000",
  ...
}
```

## Error Responses

When an error occurs, the API returns an error object with appropriate HTTP status code:

<ResponseField name="error" type="string" required>
  Human-readable error message describing what went wrong
</ResponseField>

<ResponseField name="code" type="integer" required>
  HTTP status code
</ResponseField>

<ResponseField name="details" type="object">
  Additional error details when available (validation errors, missing fields, etc.)
</ResponseField>

### Common Error Codes

<AccordionGroup>
  <Accordion title="400 Bad Request" icon="circle-xmark">
    Invalid request parameters or malformed input.
    
    ```json
    {
      "error": "Invalid block height: must be a positive integer",
      "code": 400,
      "details": {
        "field": "height",
        "provided": "abc"
      }
    }
    ```
    
    **Common causes:**
    - Invalid parameter types
    - Missing required fields
    - Malformed JSON
    - Out-of-range values
  </Accordion>
  
  <Accordion title="401 Unauthorized" icon="lock">
    Missing or invalid API key.
    
    ```json
    {
      "error": "Missing or invalid API key",
      "code": 401
    }
    ```
    
    **Solution:** Include a valid API key in the `X-API-Key` header.
  </Accordion>
  
  <Accordion title="404 Not Found" icon="magnifying-glass">
    Requested resource does not exist.
    
    ```json
    {
      "error": "Account not found",
      "code": 404,
      "details": {
        "account_id": "nonexistent.near"
      }
    }
    ```
    
    **Common causes:**
    - Account doesn't exist
    - Transaction hash not found
    - Invalid block height
    - Collection contract not deployed
  </Accordion>
  
  <Accordion title="429 Rate Limit Exceeded" icon="gauge-high">
    Too many requests in a given time period.
    
    ```json
    {
      "error": "Rate limit exceeded",
      "code": 429,
      "details": {
        "limit": 100,
        "window": "1 minute",
        "retry_after": 45
      }
    }
    ```
    
    **Solution:** Implement exponential backoff or reduce request frequency.
  </Accordion>
  
  <Accordion title="500 Internal Server Error" icon="triangle-exclamation">
    Unexpected server error.
    
    ```json
    {
      "error": "Failed to fetch account data",
      "code": 500
    }
    ```
    
    **Solution:** Retry the request. If the problem persists, contact support.
  </Accordion>
</AccordionGroup>

## Rate Limits

<Warning>
  Rate limits are enforced per API key. Exceeding limits will result in `429` errors.
</Warning>

Current rate limits:

| Tier | Requests per Minute | Requests per Hour |
|------|---------------------|-------------------|
| Free | 60 | 1,000 |
| Pro | 600 | 10,000 |
| Enterprise | Custom | Custom |

**Best Practices:**
- Cache responses when possible
- Use pagination efficiently
- Implement retry logic with exponential backoff
- Batch requests when available (e.g., transaction lookup)

## Data Freshness

Different endpoints have different latency characteristics:

| Data Type | Typical Latency | Notes |
|-----------|----------------|-------|
| Blocks | ~2 seconds | Near real-time |
| Transactions | ~2 seconds | Indexed as blocks arrive |
| Accounts | ~5 seconds | Balance updates may lag |
| Transfers | ~10 seconds | Requires event parsing |
| Graph Data | ~30 seconds | Relationship updates are batched |

<Info>
  For real-time critical applications, consider using the JSON RPC endpoint which queries the blockchain directly.
</Info>
